1. Spring Security는 현업에서 어떻게 쓰일까? (기본 흐름 vs 현업의 아키텍처)

슬라이드 1

시각 자료 : 일반적인 시큐리티 로그인 필터 체인 다이어그램에 DB 접근 부분에 빨간색 경고 표시를 한 그림

스크립트 : Spring Security의 가장 기본적인 로그인 과정에 대해서 살펴보려고 합니다. 사용자가 로그인을 시도하면 앞서 설명드린대로 DaoAuthenticationProvider라는 객체가 UserDetailService를 호출해서 사용자 정보를 가져오고 검증을 합니다. 

이 과정은 Security가 제공하는 Web Filter 단에서 일어납니다. 그런데 여기서 내부적으로 유저 조회로직이 존재하게 되는데 해당 내용은 계층 간의 역할을 엄격히 분리하는 현대적인 아키텍처의 원칙에 어긋나게 됩니다.

슬라이드2

시각 자료 : Security의 Filter 로그인을 x 표시 후 Controller → Service → DB로 이어지는 클린아키텍처 구조도 옆에 BcryptPasswordEncoderAdapter가 외부 인프라로 빠져 있는 모습

스크립트 : 실제로 Spring 공식 문서에서도 비즈니스 로직에 대한 검증은 계층형 아키텍처에서 웹 계층은 여러 계층 중 하나일 뿐이며, 서버 측 애플리케이션의 진입점 역할을 하고 서비스 계층에 정의된 서비스 객체(퍼사드)에 비즈니스 관련(그리고 프레젠테이션 기술에 구애받지 않는) 사용 사례를 처리하도록 위임합니다. 그래서 구현시 security에서 제공하는 로그인 필터를 사용하지 않고 하나의 usecase로 보고 비즈니스 로직을 서비스 로직 안에서 직접 구현했습니다. 마찬가지로 oauth2로그인도 filter를 타지 않고 argument resolver를 통해서 컨트롤러의 요청을 받은 뒤에 서비스 계층에서 db에 접근을 하도록 통제했습니다.
여기서 security의 암호화 기능은 클린 아키텍처의 원칙에 따라서 핵심 로직은 시큐리티를 몰라도 되기에 외부 인프라로 빼서 사용했습니다. 도메인이 비밀번호 검증해줘라고 포트를 통해 던지면 밖에서 security가 처리해 주는 방식으로 바꿨습니다.

슬라이드 3 : 그럼 현업에서 security는 언제 어떻게 쓰나요?

시각자료 : api 서버 앞단을 지키고 있는 (CORS, JWT, RBAC) 다이어그램

발표 스크립트 :

- 로그인 검증을 서비스 계층으로 넘겼다면 현업에선 Spring Security의 역할은 보안 인프라의 뻐대 역할을 합니다. 여기선 순수 JWT 서명 검증을 앞단에서 처리합니다. 토큰이 위조 되었는지 필터에서 빠르게 걸러내는 용도로 사용합니다. 그리고 CORS 같은 네트워크 보안 설정을 중앙 통제 합니다. 그리고 RBAC입니다. 이 사용자가 일반 유저인지 관리자인지 판별하고 API 접근을 차단하거나 허용하는 문지기 본연의 역할에 집중하는 겁니다.

슬라이드 4 : 인증 객체는 왜 필요할까?

시각 자료 : 무거운지갑(비밀번호, 개인정보)를 두고 가벼운 출입증(이메일, 권한)만 들고 다님

스크립트 : 마지막으로 저번에 daoprovider에서 인증 객체를 만들어서 반환을 한다고 했었는데 이때 이 인증 객체에는 인증 전에는 이메일 비밀번호에서 인증 후에는 비밀번호를 폐기하고 권한을 담아서 인증 객체(Authentication)을 만듭니다. 여기서 이 객체는 서버 어디서든 꺼내볼 수 있는 전역 신분증 역할을 하기 때문에 컨트롤러나 서비스 로직 어디서든지 db 조회 없이 이 신분증만 보면 되기에 이 인증 객체가 필요하고 spring security가 제공하는 안전한 상태 관리 기법입니다. 

1. **현업에서 OAuth2 쓰기 (with Keycloak)**

슬라이드1 : OAuth2의 내부 동작 흐름과 콜백의 이해

시각 자료 : 사용자가 내 서버에 접근 → 인증 서버(구글)로 리다이렉트 → 인증 후 내 서버의 콜백 주소로 인가 코드 전달 → 서버 간 토큰 교환 흐름도

스크립트 : OAuth2의 핵심은 신원 확인의 위임입니다. 내부 동작 흐름인 인가 코드 부여 방식을 보면 사용자가 로그인을 시도하면 spring은 사용자를 인증 서버의 로그인 페이지로 리다이렉트 시킵니다.

구글에서 로그인을 마치면 구글은 사용자를 우리 서버의 특정 주소 callback 주소로 돌려보냅니다. 이때 임시로 인가 코드라는걸 쥐어줍니다.

spring security는 이 콜백을 인터셉트 해서 사용자의 브라우저를 거치지 않고 서버 대 서버 통신으로 구글에 코드를 전달하고 access Token을 받아옵니다.

슬라이드 2 Keycloak이 없을 때의 oauth2 흐름

시각 자료 : 내 서버 하나에 카카오, 네이버, 구글, 애플 등등 수 많은 인증 서버가 복잡하게 얽혀 있는 다이어그램

발표 스크립트 : 

만약 이 keycloak없이 순수security만으로 모두 구현을 하게 된다면 모든 기관의 callback 처리를 개별적으로 처리해야합니다. 구글이 주는 정보 형식과 카카오가 주는 형식이 달라서 코드가 쌓이게 됩니다.

슬라이드 3 : 현업의 해결책, 통합 인증 센터

시각 자료 : 내 서버 - Keycloak - 구글, 카카오, 네이버 로 이어지는 구조

스크립트 : 

이러한 문제를 해결하기 위해서 keycloak이라는 사설 통합 인증 센터를 도입합니다. 이를 도입하게 되면 security의 역할이 크게 줄어듭니다. 카카오와 구글이 직접 대화하지 않고 오직 keycloak과만 통신하면 되고 콜백의 주체도 바뀌게 됩니다. 사용자가 구글 로그인을 누르게 되면 구글은 keycloak로 콜백을 보내게 됩니다. 키클록이 구글, 카카오의 제각각인 응답을 하나의 표준화된 OIDC라는 국제 표준 규격의 json으로 포장한 후 spring에 한번 콜백을 보냅니다. 즉 이 키클록이 외부 인증 작업을 다 처리해주고 스프링은 키클락이 발행한 jwt만 검증하면 되는 관심사 분리가 이루어집니다.

슬라이드 4 : 공식 문서 기반 spring은 어디까지, 비즈니스 로직은 어디에..?

시각자료 : spring security의 oauth2 콜백 필터와, 이를 거쳐 커스텀 아규먼트 리졸버 → 서비스 계층의 db 접근 흐름도

스크립트 : 

마지막으로 시큐리티의 공식문서를 보면 시큐리티는 사용자의 속성 데이터를 가져와서 그것을 인증된 객체로 반환하는 것 까지 합니다. 즉 인가 코드를 토큰으로 교환하는 프로토콜 통신까지만 책임지는 것을 기본으로 합니다. 

기본 설정대로라면 토큰을 받은 직후 OAuth2UserService 라는 곳에서 강제로 유저 정보를 세션에 밀어넣습니다. 하지만 db 접근은 비즈니스 로직에서 처리해야 한다고 설계했습니다. 그래서 시큐리티는 keycloak로부터 토큰을 받아오는 인프라적인 역할까지만 하고 컨트롤러단에서 리졸버를 통해서 토큰 정보만 뺴낸 뒤에 서비스 계층으로 넘겨서 db 조회를 하고 새로운 회원이면 가입을 시키도록 했습니다.

1. CA란? MVC랑 뭐가 다를까?

슬라이드1 : 기존 MVC 패턴의 한계

시각 자료 : 선이 복잡하게 얽힌 mvc 다이어그램

스크립트 : 가장 먼저 접하고 배우게 되는 mvc 패턴은 빠르게 만들기에 좋지만 공간이 분리되지 않아서 점점 프로젝트가 커지게 되면 관리하기가 힘들게 되는데요. 예를 들어서 MVC로 개발하다보면 핵심 비즈니스 로직 안에 db 쿼리나 웹 통신 코드가 섞여서 들어갈 수 있습니다. 이런 상황에서 갑자기 db를 mysql에서 oracle로 바꿔야하는 상황이나 웹 대신 모바일 전용의 api로 바꿔야 하는 요구사항이 생기게 된다면 mvc에 상황에선 대처하기가 힘들고 기존의 코드를 뜯어고쳐야 하는 상황이 생깁니다.

슬라이드2 : 클린아키텍처 방을 구성하다

시각 자료 : auth에 대한 클린아키텍처 그림

스크립트 : 이런 한계점을 클린 아키텍처가 해결해줍니다. 저는 프로젝트를 6개의 모듈로 쪼개고 CA를 적용했습니다. 이 구조의 절대 규칙은 하나입니다. 의존성은 무조건 바깥에서 안쪽으로만 향하게 한다. 바깥은 안쪽을 호출할 수 있지만 안쪽은 바깥에 뭐가 있는지 전혀 몰라야합니다. 가장 깊은 곳에 domain과 application 모듈은 순수한 자바로 짜여진 코드이고 외부 웹 요청은 presentation이 db나 외부 통신은 infra라는 모듈에서 대신 처리하고 결과만 반환합니다.

슬라이드3 : 매력적인 CA의 위력

시각 자료 : 버그 발생 시 Presentation인지 infra인지 바로 찾기 가능한 이미지

스크립트 : 이 CA라는게 정말 좋은 구조라고 느꼈던게 일단 오류가 나더라도 어느 부분에서 수정이 이루어져야 하는지 예측이 가능하다라는게 좋았구요 예를 들어서 http부분은 presentation에서 확인하면 되고 db는 infra이렇게 예측이 가능하다라는 점이 있구요 조립을 bootstrap이라는 곳에서 조립을 하고 설정값을 관리하다보니까 전체 시스템의 연결 상태가 한곳에서 관리된다는게 좋았습니다. 형이 이때 이 설정 값을 3개씩 6개로 18개를 관리하라고 해서 이게 처음에 좀 와닿지가 않았는데 외부의 관리 포인트들이 늘어나고 해당 인프라나 presentation을 관리하기 위해서 해당 모듈에서 설정 파일을 각각 관리하고  bootstrap에서 합치는게 가독성 측면이나 유지보수 측면에서 그런 구조가 더 나을 수 있겠다라는 생각을 했습니다.

슬라이드4 : 아키텍처의 붕괴를 막는 파수꾼, 그것이 나 ArchUnit

시각 자료 : 코드를 올렸을 때 규칙 위반 Domain이 infra를 참조했습니다. 라는 경고를 띄우는 자동화 테스트 화면

발표 스크립트 : 그리고 형이 이 아키텍처를 구성을 하면서 어떤 사람이 와서 같이 개발을 해도 해당 위치에서 개발이 되어야 한다라고 하셨잖아요 근데 이 아키텍처에 대해서 좀 완벽하게 모르는 사람이 개발을 하다가 의존성 실수를 할 수도 있고 네이밍 실수도 할 수도 있고 또한 순수성을 해칠수도 있는데 이를 현업에선 어떻게 해결을 할까해서 찾아봤는데 ArchUnit이라는 라이브러리가 이를 해결해주고 또 이를 많이 사용하는 것 같았습니다. 그래서 이 테스트 유닛을 적용해서 의존성은 application 계층은 presentation이나 infra를 참조할 수 없다 또는 application 같은 경우는 usecase로 네이밍이 반드시 끝나야된다 등을 명시해줌으로써 이를 강제하도록 했습니다. 

1. 서버와 db-migration

슬라이드 1 : 환경 분리

시각자료 : 좌측 자전거, 자동차, 레이싱카 셋팅이 다름을 보여주는 아이콘

- 급격하게 치솟는 트래픽 그래프, 이를 안정적으로 처리해 내는 서버의 테스트 결과 화면

스크립트 : 그리고 CA를 구성하고 나서 서버를 실행을 할때 저는 로컬환경, dev환경, prod 환경을 dev 환경은 노트북에서 실행되는 환경이라고 가정하고 데스크탑은 prod 환경이라고 가정하고 진행했습니다. 이 과정에서 핵심이 되는 목표는 대용량 트래픽과 대용량 데이터를 감당할 수 있는 구조를 만드는 것이었습니다. 그래서 각 환경에 맞춰서 yml 설정 파일을 분리하고 운영 환경에서는 트래픽 병목을 막기 위해서 DB 커넥션 풀을 튜닝하고 설령 장애로 인해 서버가 죽더라도 사용자의 요청은 안전하게 끝마치고 종료되는 설정을 추가했습니다. 현재 가용한 데스크탑 리소스의 한계 내에서 실제로 순간적으로 폭증하는 스파이크 트래픽을 발생시켜 봤습니다. 이를 안정적으로 소화하고 우아하게 종료되는지 확인까지 했구요.

슬라이드 2 : DB의 버전을 관리하다. Flyway 도입기

시각 자료 : 우측 블록이 차곡차곡 쌓이면서 db가 완성되는 그림

좌측 하단 k3s 환경에서 flyway job이 먼저 실행되어 db를 셋팅하고 난 뒤, 메인 서버 pod가 켜지는 깔끔하게 분리되는 배포 흐름도

스크립트 : 기존의 flyway의 사용은 제가 서버가 시작이 될 때 해당 flyway를 실행시켜서 독립적으로 실행을 시키기 못했습니다. 처음엔 이를 분리하고자해서 db가 시작할 때 설정 파일들을 가져오게 해서 붙이도록 했었는데 이렇게 했을때의 문제가 기존에 있는 auth 서버가 띄워질 때 관련 설정 bean들이 필요하게 된다라는 점이었습니다. 의존도가 auth에 높기 때문에 그래서 이를 더 깔끔하게 해결을 못하나 고민을 하다가 해당 라이브러리를 완전히 분리를 통해서 관리하도록 하면 완전 별도의 라이브러리로 관리가 되면서 서버 애플리케이션에 영향을 주지 않게 할 수가 있었습니다. 이를 빌드 이미지를 만들 때도 별도의 라이브러리를 복사해서 해당 애플리케이션을 k3s에 올릴때 별도의 job으로 실행이 될 수 있도록 했습니다.

1. Kube를 k3s vs K8s로 구성하기

슬라이드 1: 노트북에 데이터센터 짓기(Multipass)

시각자료 : 노트북안에 3대의 독립된 우분투(Control Plane 1, Worker 2)가 떠 있는 아키텍처 다이어그램

스크립트 : 많은 서버들을 운영하려면 물리적인 컴퓨터가 여러 대 필요하게 되는데요 우분투에서는 기본적으로 가상환경을 구성할 때 멀티패스라는걸 제공하기 때문에 이를 하용해서 노트북에 독립된 우분투 가상환경을 만들었구요 저는 이 3곳의 가상환경에 control plane 노드 1개와 worker node 2대로 나누어 진행을 했습니다.

슬라이드2 : K8s vs K3s 왜 가볍지?

시각자료 : 거대한 컨테이너 화물선(K8s)과 그옆을 빠르게 지나가는 소형(k3s) 그리고 K3s 구조 내부 구조도

스크립트 : k8s와 k3s는 이러한 가상컴퓨터 환경을 통제하는 도구인데요 이 k3s라는 도구가 더 리소스를 덜 먹는 이유는 k8s에 들어가는 클라우드 연동 코드들이 없구요 쿠버네티스는 기본적으로 노드라는 공간에 여러 리소스가 존재하고 해당 리소스들이 서로 통신을 하는 구조입니다. 그래서 아까 설명한 이 control plane과 worker node로 분리를 해서 설명을 했지만 쿠버네티스 입장에서 이 노드들은 다른게 아니라 같은 노드고 이 control plane안에 etcd라는 kubernetes 내부에서 노드끼리 서로 통신하는 과정에서 생기는 데이터를 저장하고 관리하는 저장소입니다. 간단하게 말해서 이게 있고 없고의 차이가 control plane 노드고 아니고를 구분하게 되는데 이 etcd에서 사용하는 무거운 db 대신에 가벼운 sqlite를 k3s에서 사용하구요 패키징을 통해서 복잡한 구성 요소들을 100MB파일 하나로 압축했습니다.

슬라이드 3 : K3s의 편리함, 편안함에 속지말아요

시각자료 : K3s 안에 기본 탑재된 Traefik, Flannel/Calico, Local Storage 아이콘들

스크립트 : K3s의 또 다른 장점은 올인원이라는건데 서버간의 통신을 하려면 네트워크 플러그인(CNI)이 필요한데 K3s는 이를 알아서 깔아줍니다. 형이 저번에 아치 리눅스 구성을 해보라고 하셨잖아요. 우분투와 달리 진짜 얘는 아무것도 뭔갈 안해주니까 이런걸 다 직접 해보기엔 이런 환경이 근본을 알기에 적합할 거 같다고 생각했습니다. 물론 아직 여기서 인프라 구축까진 안해봤고 지금은 운영체제 설치랑 개발환경 셋팅 진행하고 있습니다. 지금 우분투 환경에서 했던거를 여기서 다시 하게 되면 이런 내부 원리를 알 수 있게 되지 않을까 싶네요.

슬라이드 4 : 현업에서 K3s 쓰긴하나?

시각 자료 : 공장, 편의점 포스기, IoT 기기등 엣지 디바이스에 설치된 K3s 다이어그램

스크립트 : 현업에서 K3s를 쓰기보단 그냥 공부용인줄만 알았는데 공식 문서에 들어가보니까 iot에 이 k3s가 많이 활용된다는 걸 볼 수 있었구요 공장의 스마트로봇에서도 이를 활용한다는걸 알수 있었습니다.

1. Env 숨기기 작전(Secret → Sealed Secret → Vault)

슬라이드1 : 1단계 진화 - 쿠버네티스 기본 Secret의 함정

시각 자료 : 일반 텍스트가 Base64로 변환되는 과정과, 그게 다시 쉽게 원문으로 복구되는 것을 보여주는 그림

스크립트 : 여기서 가장 큰 고민은 k3s 인프라 위에서 띄울 때 이 env 내용중에 민감한 내용을 어떻게 숨길 것인가에 대한 문제입니다. 가장 먼저 시도할 수 있는 방법은 쿠버네티스가 기본적으로 제공하고 있는 kube secret을 사용하는 것입니다. 하지만 이 방식은 데이터를 암호화하는 방식이 아니라 단순히 base64로 인코딩을 하는 방식이기 때문에 누구나 열어볼 수 있다는 단점이 있습니다. 

슬라이드 2 : 비대칭키를 활용한 Sealed Secret

시각 자료 : 깃허브에 올라간 코드는 자물쇠로 잠겨있고 오직 k3s 클러스터 내부에서만 열쇠로 이를 풀어볼 수 있는 아키텍처 다이어그램

스크립트 : 이런 내용들이 git에도 올라갈텐데 이를 숨기고자 도입한게 sealed secret였습니다. 공개키와 개인키를 활용하는 비대칭 암호화 기술을 쿠버네티스에 적용한 건데요. 이 방식을 사용하면 민감한 내용을 암호화해서 개인키로만 풀 수 있기에 유출이 되더라도 비밀번호를 알아낼 수 없는 1차 구조를 만들었습니다. 

슬라이드3 : 중앙 집중형 시크릿 관리, Vault

시각 자료 : 서버가 구동될 때 vault agent/injector가 금고에서 비밀키를 꺼내서 서버 내부에 파일 형태로 꽂아주는 주입 흐름도

수많은 비밀번호 대신 하나의 unseal 토큰만 존재하는 다이어그램

스크립트 : 저는 기본적으로 이부분을 공부하기 전까지 런타입에 직접 주입을 하는 방식을 사용했었는데 이렇게 사용을 하게 되면 실행중인 서버에 env를 쳐서 보게 되거나 찍어보면 전부 다 나오게 되는 걸 보고 쉽게 여기까지 올 수만 있다면 중요한 값들은 가져갈순 있겠다라고 생각했구요 그래서 이 vault에서 제공하는 agent와 injector를 사용해서 파일 기반 주입을 하게 되면 해당 내용이 안남게 됩니다. 그리고 그 과정에서 vault가 기본적으로 sealed된 상태인데 이를 unseal 하기 위해서 운영자가 직접 해줘야하는데 그 과정에서 이 roottoken이 너무 쉽게 노출이 되기 때문에 해당 토큰을 최소로 사용하고 없애고 unseal하는 과정은 vault provider에 위임을 하였습니다. 이렇게 구조를 변경하게 되면 운영자가 직접 unseal해야되는 빈도는 줄어들게 되구요 vault provider에서는 vault-workload의 unseal과 통신에 필요한 token관리만 하면 되기에 노출되는 이 토큰의 권한을 최소한으로 낮출 수 있다는 장점이 있습니다. 다만 그 과정에서 이 토큰을 다시 kubernetes에 secret에 저장해야만 했는데 이게 또 여기에 저장이 된다는게 좀 딜레마가 됬었는데 이 토큰을 여기서 관리를 하는게 최선인지 현업에선 어떤방식으로 이를 관리하는지에 대해서 좀 여쭤보고 싶었습니다. 클라우드를 사용하게 되면 kms를 쓰겠지만 무조건 kms를 쓰진 않을거 같은데 보안이 중요시되는 곳이라면 kms가 아니라 다른 상위 시크릿스토어를 구축할 거 같은 느낌이어서.. 다만 여기서 제가 이렇게 하고 넘어갔던 이유는 토큰의 권한도 최소한으로 가지고 있고 이 토큰이 탈취가 되어도 vault를 unseal하는 권한 밖에 없으니까 표출되는 표면적을 최소화 시켰기에 괜찮다고 판단했습니다.
